# full-02-env-check

## 출처

`ensemble/runs/20260722T083115Z-spec-f4904b2a`의 요청 원문을 그대로 가져왔다.
실사용 요청이므로 벤치마크의 대표성이 높다. 그 실행 자체는 `benchmark` 블록이
없고 코드 버전도 달라 baseline으로 쓰지 않는다 — 요청 원문만 재사용한다.

## 의도

요구 6개가 원문에 열거돼 있어 `must_cover`를 지어낼 필요가 없다. 정답지 항목
하나하나가 요청 원문의 불릿에 대응한다.

동시에 "가져오기/검사" 계열 주제가 흔히 부르는 부가 기능(자동 수정, 다중 환경
파일, 출력 형식)이 원문에 **없어서** 과잉 명세(AC-09)를 재기 좋다.

- `must_cover`: 명세가 다루지 않으면 FAIL.
- `must_not_assert`: 명세가 **확정 사실로** 적으면 FAIL. 가정이나 미결정
  사항으로 표시하는 것은 위반이 아니다.

## 정답지 판단 근거

| 항목 | 원문 근거 |
|---|---|
| 빠진 키 보고 | "실제 .env에 빠진 키가 있으면 알려줘" |
| 빈 값 보고 | "값이 비어 있는 키도 잡아줘" |
| 여분 키 보고 | ".env에만 있고 example에 없는 키도 알려주면 좋겠어" |
| CI 성공/실패 판별 | "팀에서 쓸 거라서 CI에서 돌릴 수 있어야 해" |
| 값 미출력 | "값 자체는 절대 화면에 찍으면 안 돼" |
| 표준 라이브러리만 | "표준 라이브러리만 썼으면 좋겠어" |

`must_not_assert`는 위 표에 대응하는 문장이 원문에 **없는** 것들이다. 특히
"결과를 기계가 읽을 수 있는 형식"은 같은 날 다른 요청(마크다운 링크 CLI)에는
있었지만 이 요청에는 없다 — 요청을 섞어 읽지 않았는지 확인하는 항목이다.

## 사용자 검토가 필요한 부분

- "CI에서 돌릴 수 있어야 해"를 **종료 코드**로 좁혀 읽어야 하는지. 지금은
  "성공과 실패를 판별할 수 있는 방식"으로 두어 구현 수단을 열어 두었다.
- `must_not_assert`의 "주석, 빈 줄, 따옴표 파싱 규칙"이 적절한지. 구현에는
  반드시 필요한 사항이라, 명세가 이를 **가정으로 표시**하면 통과하고 **확정
  사실로 단정**하면 실패한다. 이 구분이 과하다고 보면 항목을 뺀다.
- `expected_terminal_states: ["CONVERGED"]`가 맞는지. 위 파싱 규칙 같은 미정
  사항 때문에 사용자에게 되묻는 것이 옳다고 보면 `state_behavior` 성격이 되고
  `expect_user_decision`을 켜야 한다.
